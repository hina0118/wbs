# GitHub Copilot レビュー観点まとめ

本ドキュメントは、過去の PR で GitHub Copilot から指摘された内容をレビュー観点として整理したものです。  
新規 PR 作成時やコードレビュー時のチェックリストとして活用してください。

**参照元**: PR #21, #59, #60, #62, #69, #70, #73, #74, #75, #76 の Copilot レビュー対応計画・分析ドキュメント

---

## 1. セキュリティ

### 1.1 機密情報・個人情報の取り扱い

| 観点         | 例                                                     | 対応例                                                                                        |
| ------------ | ------------------------------------------------------ | --------------------------------------------------------------------------------------------- |
| **ログ出力** | メール本文・ペイロード・件名がデバッグログに記録される | 本番ではデバッグログ無効化、機密情報のマスキング、`info` → `debug` への格下げ                 |
| **API キー** | URL クエリパラメータに API キーが含まれる              | ログ出力時にマスキング（`safe_url` 等）、仕様上ヘッダー認証が使えない場合はリスク許容を文書化 |

### 1.2 入力検証・パストラバーサル

| 観点                       | 例                                             | 対応例                                                                                            |
| -------------------------- | ---------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| **URL 検証**               | 外部 URL を未検証で fetch（SSRF 攻撃）         | `https` のみ許可、プライベート IP・localhost ブロック、`validate_image_url` 相当の検証            |
| **パストラバーサル**       | `file_name` を未検証で `join()` に使用         | `file_name` が単一ファイル名（`/`、`\`、`..` を含まない）であることを検証する `is_safe_file_name` |
| **assetProtocol スコープ** | `$APPDATA/**` でアプリデータ全体にアクセス可能 | `$APPDATA/jp.github.xxx/images/**` など必要最小限に制限                                           |

### 1.3 リソース制限

| 観点                 | 例                                         | 対応例                                                  |
| -------------------- | ------------------------------------------ | ------------------------------------------------------- |
| **画像サイズ**       | Content-Length や body のサイズ制限なし    | 10MB 等の上限を設け、超えたらリクエスト拒否             |
| **画像フォーマット** | 任意形式を保存可能（マルウェア混入リスク） | `image` クレートで JPEG/PNG/WebP のみ許可               |
| **ZIP 読み込み**     | `read_to_string` でサイズ上限なし          | `entry.size()` で上限チェック、巨大 ZIP による DoS 対策 |

## 2. エラーハンドリング・堅牢性

### 2.1 Rust

| 観点                 | 例                                       | 対応例                                                                  |
| -------------------- | ---------------------------------------- | ----------------------------------------------------------------------- |
| **unwrap() / panic** | Mutex ロック失敗時に `unwrap()` で panic | `match` で `Ok`/`Err` を明示的に処理、失敗時はログ出力して継続          |
| **型変換**           | `as usize` で桁あふれの可能性            | `clamp_batch_size` 相当の安全変換、`usize::try_from` で失敗時は default |
| **Base64 デコード**  | 失敗時に元データをそのまま使用           | 適切なフォールバックとエラーハンドリング                                |
| **Err の捨て方**     | 固定文言を返して原因切り分けが困難       | `Err(e.to_string())` をそのまま返す                                     |

### 2.2 TypeScript / React

| 観点                     | 例                                       | 対応例                                                                     |
| ------------------------ | ---------------------------------------- | -------------------------------------------------------------------------- |
| **エラー型アサーション** | `err as string` による型強制             | 型ガードを使用した安全なエラーハンドリング                                 |
| **API 戻り値型**         | `bool` のまま（エラー時は throw すべき） | `Result<bool, String>` に変更、フロントの catch で `false` 設定            |
| **Mock の型**            | `onerror` の型定義が実際の API と不一致  | `(this: FileReader, ev: ProgressEvent<FileReader>) => void` 形式に合わせる |

## 3. 非同期・リソース管理

| 観点                           | 例                                                             | 対応例                                                               |
| ------------------------------ | -------------------------------------------------------------- | -------------------------------------------------------------------- |
| **setTimeout クリーンアップ**  | `useEffect` 内の `setTimeout` にクリーンアップなし             | `useEffect` の return で `clearTimeout`                              |
| **Promise の未処理 rejection** | `notify()` を await していない                                 | `await notify(...)` または `void notify(...).catch(...)` で明示      |
| **古いリソースの削除**         | ON CONFLICT 更新時に古い `file_name` の画像が残る              | 更新前に既存 file_name 取得、DB 更新後に古いファイル削除             |
| **early return 時の finish()** | `try_start()` 成功後の early return で `finish()` が呼ばれない | 全 early return 経路で `finish()` を呼ぶ、または RAII ガード型を導入 |

## 4. データベース・マイグレーション

| 観点                         | 例                                           | 対応例                                                                         |
| ---------------------------- | -------------------------------------------- | ------------------------------------------------------------------------------ |
| **マイグレーションの整合性** | PR 説明に「003 を追加」とあるが 001 のみ変更 | 001 に集約するか、003 を実際に追加。既存 DB への適用方針を PR 説明に明記       |
| **既存 DB 互換性**           | 新カラム追加が既存 DB に反映されない         | 001 は CREATE TABLE IF NOT EXISTS のため新規のみ。既存向けは別マイグレーション |
| **UNIQUE 制約**              | 既存 DB に古い UNIQUE が残り競合             | 001 への集約、既存 DB マイグレーション方針をドキュメント化                     |
| **一時テーブル**             | `temp_filter_ids` を明示的に DROP していない | `tx.commit()` 前に `DROP TABLE IF EXISTS` を追加                               |

## 5. React / フロントエンド

### 5.1 レンダリング・キー

| 観点                    | 例                                               | 対応例                                                                     |
| ----------------------- | ------------------------------------------------ | -------------------------------------------------------------------------- |
| **key に index を使用** | `key={index}` で再レンダリング時に不整合         | 一意の ID があれば `key={row.id}`、なければ `key={\`${url}-${index}\`}` 等 |
| **onError 無限ループ**  | 画像読み込みエラーでフォールバックが繰り返し発火 | `data-error-handled` で 1 回のみフォールバック                             |

### 5.2 アクセシビリティ

| 観点             | 例                                              | 対応例                                     |
| ---------------- | ----------------------------------------------- | ------------------------------------------ |
| **aria-label**   | ボタンにスクリーンリーダー用の説明がない        | `aria-label="画像を検索"` 等を追加         |
| **Space キー**   | ボタンで Space 押下時に `preventDefault` がない | `onKeyDown` で `e.preventDefault()` を追加 |
| **ファイル入力** | `id` / `aria-label` が不足                      | アクセシビリティ属性を追加                 |

### 5.3 useEffect・依存配列

| 観点                     | 例                                                   | 対応例                                 |
| ------------------------ | ---------------------------------------------------- | -------------------------------------- |
| **依存配列の過剰**       | コールバック内で未使用の `onOpenChange` を依存に含む | 未使用なら依存配列から削除             |
| **useEffect の重複実行** | filterLevel 変更時に loadLogs が 2 回呼ばれる        | 統合または条件分岐で不要な再実行を防ぐ |

## 6. コード品質・一貫性

| 観点                       | 例                                                                 | 対応例                                         |
| -------------------------- | ------------------------------------------------------------------ | ---------------------------------------------- |
| **API 一貫性**             | 他コマンドは `Result` を返すのに当該のみ `bool`                    | 他と揃えて `Result<T, E>` に統一               |
| **関数シグネチャ**         | `_app_data_dir` の有無が他と異なる                                 | 共通パターンに揃える                           |
| **ロジックの重複**         | order-item-card と order-item-row で同じ条件分岐                   | `getProductMetadata` を utils に抽出して共通化 |
| **コメントと実装の不一致** | コメント「order_number のみで再検索」と実装が違う                  | コメントを実装に合わせて修正                   |
| **Doc コメント**           | 関数名が挙動と一致していない（例: clamp だが上限はクランプしない） | Doc で「上限はクランプしない」等を明記         |
| **未使用インポート**       | `DialogOverlay` が未使用                                           | インポートを削除                               |
| **マジックナンバー**       | ハードコードされた数値                                             | 定数化とコメント追加                           |

## 7. テスト

| 観点               | 例                                                           | 対応例                                               |
| ------------------ | ------------------------------------------------------------ | ---------------------------------------------------- |
| **テスト不足**     | 新機能・新コンポーネントにテストがない                       | 主要分岐・エッジケースをカバーするテストを追加       |
| **モックの競合**   | setup で一律モックすると個別テストのアサーションが成立しない | setup でモックを export し、テストから参照           |
| **E2E の locator** | `message` 指定で複数要素にマッチし得る                       | `.first()` を付けるか、件数を絞る条件を追加          |
| **統合テスト**     | `apply_cancel` 等の重要ロジックに DB 状態検証がない          | 1 件減算・0 で削除・商品不一致・注文不一致等をカバー |

## 8. パフォーマンス

| 観点                 | 例                                 | 対応例                                       |
| -------------------- | ---------------------------------- | -------------------------------------------- |
| **グローバル Mutex** | ロック競合・デッドロックリスク     | Tauri のステート管理への移行を検討           |
| **ログの Mutex**     | ログ記録の度にロック取得           | 非同期チャネルやロックフリーデータ構造の検討 |
| **SQL の非効率**     | LENGTH 関数の重複計算              | CTE で計算を一元化                           |
| **メモリ読み込み**   | 画像を `fs::read` で丸ごと読み込み | サイズ上限、またはストリーミング             |

## 9. ドキュメント・PR 説明

| 観点                     | 例                                                                  | 対応例                                                                  |
| ------------------------ | ------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| **PR 説明の不足**        | リポジトリ全体に影響する変更（.gitattributes 等）が記載されていない | Changes に追記                                                          |
| **マイグレーション方針** | 既存 DB への適用方針が未記載                                        | アーキテクチャ doc に「既存 DB のマイグレーション方針」セクションを追加 |
| **設計書との整合**       | 実装と設計書の記載が食い違う                                        | 設計書を更新するか、PR コメントで意図を説明                             |

## 10. 優先度ラベル（Copilot ルール）

プロジェクトの `.github/copilot-instructions.md` に定義された優先度:

| ラベル            | 対象                                                 | 対応                                             |
| ----------------- | ---------------------------------------------------- | ------------------------------------------------ |
| **P0: Critical**  | 動作不能、致命的バグ、セキュリティ脆弱性、データ破損 | 修正必須。マージ不可                             |
| **P1: Important** | パフォーマンス低下、一貫性欠如、推奨されない古い手法 | 修正を強く推奨。状況により後回し（Issue 化）許容 |
| **P2: Nitpick**   | タイポ、変数名の好み、スタイル、微細なリファクタ     | 提案のみ。開発者の判断で無視可                   |
